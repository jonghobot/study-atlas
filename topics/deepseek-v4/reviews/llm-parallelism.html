LLM Parallelism: 거대 모델을 나눠 학습하고 서빙하는 방법

LLM parallelism은 “GPU를 많이 붙인다”가 아니라, 무엇을 어떤 축으로 나눌 것인가를 정하는 설계 문제입니다. 파라미터, activation, sequence, expert, KV cache, request batch를 서로 다른 방식으로 쪼개야 trillion-scale 학습과 million-token serving이 가능합니다.

30초 요약

LLM 병렬화는 크게 학습 병렬화와 추론/서빙 병렬화로 나눠 봐야 합니다. 학습에서는 모델 파라미터·gradient·optimizer state·activation을 여러 device에 나눠 메모리 한계를 넘고, 추론에서는 prefill·decode·KV cache·동시 요청을 나눠 latency와 throughput을 맞춥니다.

DeepSeek V4 같은 MoE/장문맥 모델을 이해할 때 병렬화는 주변 시스템 지식이 아니라 핵심 개념입니다. MoE는 expert parallelism 없이는 커지기 어렵고, 1M context는 sequence/context parallelism과 KV cache 관리 없이는 제품 latency가 무너집니다.

왜 병렬화가 필수인가

대형 Transformer는 파라미터만 큰 것이 아닙니다. 학습 중에는 gradient, optimizer state, activation까지 저장해야 하고, 추론 중에는 사용자별 KV cache가 context 길이에 비례해 쌓입니다.

따라서 단일 GPU 성능을 올리는 것만으로는 한계가 있습니다. 병렬화는 compute를 늘리는 동시에, 메모리와 통신을 어떤 device에 배치할지 결정하는 방법입니다.

훈련과 서빙은 병목이 다르다

학습은 대체로 큰 batch와 반복 step이 있으므로 throughput, GPU utilization, communication overlap이 중요합니다. 반면 서빙은 요청마다 길이가 다르고, prefill은 compute-heavy, decode는 memory/KV bandwidth-heavy인 경우가 많습니다.

그래서 학습에서 좋은 병렬화 조합이 그대로 서빙에 좋은 것은 아닙니다. 제품에서는 p95/p99 latency, batch scheduler, KV cache fragmentation까지 봐야 합니다.

1. Data Parallelism: 가장 기본이지만 공짜는 아니다

Data parallelism은 같은 모델 복사본을 여러 GPU에 올리고, 각 GPU가 다른 mini-batch를 처리한 뒤 gradient를 평균내는 방식입니다. 구현이 비교적 단순하고 scale-out의 기본값입니다.

장점
모델이 한 GPU에 들어가기만 하면 사용하기 쉽고, batch를 키워 throughput을 높이기 좋습니다.
한계
모델 자체가 한 GPU에 안 들어가면 해결책이 아니고, step마다 gradient all-reduce가 필요합니다. 너무 큰 global batch는 최적화 품질에도 영향을 줄 수 있습니다.

2. Tensor Parallelism: 행렬 곱을 쪼개는 병렬화

Tensor parallelism은 Transformer layer 안의 큰 matrix multiplication을 여러 GPU로 나눕니다. Megatron-LM은 attention/MLP의 projection matrix를 column/row 방향으로 분할해, 단일 layer의 파라미터와 계산을 device에 분산합니다.

이 방식은 모델의 한 layer가 너무 커서 단일 GPU에 부담될 때 유용합니다. 대신 layer마다 partial result를 합치기 위한 collective communication이 자주 발생하므로, NVLink/NVSwitch 같은 빠른 intra-node interconnect와 잘 맞습니다.

# 직관: Y = XW를 W = [W1, W2]로 나눔
GPU0: Y1 = X @ W1
GPU1: Y2 = X @ W2
concat(Y1, Y2) 또는 다음 row-parallel layer에서 reduce

3. Pipeline Parallelism: layer를 stage로 나누기

Pipeline parallelism은 모델의 앞쪽 layer는 GPU0, 중간 layer는 GPU1, 뒤쪽 layer는 GPU2처럼 stage를 나눕니다. GPipe는 mini-batch를 micro-batch로 쪼개 pipeline을 채워 idle time을 줄이는 대표 접근입니다.

개념의미주의점
Stage연속된 layer 묶음stage별 계산량이 불균형하면 느린 stage가 전체를 막음
Micro-batchpipeline을 채우기 위해 mini-batch를 더 작게 분할너무 작으면 kernel 효율이 떨어질 수 있음
Bubblepipeline 앞뒤에서 GPU가 비는 시간micro-batch 수를 늘리거나 스케줄을 바꿔 줄임

Pipeline은 node 간 분산에 유리하지만, activation을 stage 사이에 전달해야 하고 bubble을 완전히 없애기 어렵습니다.

4. ZeRO/FSDP: 모델 상태를 shard하기

ZeRO 계열은 data parallel 복제에서 낭비되는 메모리를 줄입니다. 일반 data parallel은 각 GPU가 parameter, gradient, optimizer state를 모두 갖고 있지만, ZeRO는 이 상태들을 GPU들에 나눠 저장합니다.

  1. ZeRO-1: optimizer state를 shard합니다.
  2. ZeRO-2: optimizer state와 gradient를 shard합니다.
  3. ZeRO-3/FSDP: parameter까지 shard하고, 필요한 순간 all-gather합니다.

핵심 trade-off는 메모리 절감과 통신 증가입니다. 모델을 더 크게 만들 수 있지만, parameter를 자주 모으고 흩어야 하므로 네트워크와 overlap 설계가 중요해집니다.

5. Sequence / Context Parallelism: token 길이를 나누기

Long-context LLM에서는 batch나 layer만 나누는 것으로 부족합니다. sequence 길이가 길어질수록 attention과 activation, KV cache가 커지기 때문에 token dimension 자체를 나눠야 합니다.

Ring Attention은 이 관점을 잘 보여줍니다. 각 device가 자기 query block을 들고 있고, key/value block이 ring을 따라 이동하면서 blockwise attention을 계산합니다. 이때 streaming softmax로 block별 결과를 합쳐 full attention 결과를 유지하려 합니다.

학습 관점
긴 sequence activation을 여러 device로 나눠 단일 GPU 메모리 한계를 넘습니다.
서빙 관점
1M context prefill을 여러 device에 펼칠 수 있지만, decode 단계의 KV 접근과 통신 latency가 별도 병목이 됩니다.

6. Expert Parallelism: MoE에서 expert를 나누기

MoE 모델은 token마다 일부 expert만 활성화합니다. 전체 파라미터는 매우 크지만 active parameter는 작게 유지할 수 있습니다. 이 구조가 커지려면 expert를 device들에 분산하고, token을 해당 expert가 있는 device로 보내는 expert parallelism이 필요합니다.

문제는 all-to-all 통신과 load balancing입니다. 특정 expert로 token이 몰리면 일부 GPU는 과부하되고, 나머지는 놀게 됩니다. 그래서 routing loss, capacity factor, expert specialization 같은 설계가 성능과 품질을 동시에 좌우합니다.

MoE 병렬화 질문왜 중요한가
expert를 몇 개 device에 나눌 것인가?메모리 배치와 통신량이 결정됨
token routing을 얼마나 균등하게 만들 것인가?품질과 load balance의 trade-off
expert parallel과 tensor/data parallel을 어떻게 섞을 것인가?대규모 cluster 효율을 좌우

7. Inference / Serving Parallelism: 요청을 어떻게 섞을 것인가

서빙에서는 학습과 다른 문제가 나타납니다. 사용자의 prompt 길이와 output 길이가 다르고, prefill과 decode의 성격이 다릅니다. Prefill은 긴 prompt를 한 번에 읽는 compute-heavy 작업이고, decode는 매 step마다 새 token 하나를 만들며 KV cache를 읽는 memory-heavy 작업입니다.

  • Continuous batching: 끝난 요청은 빠지고 새 요청이 들어오도록 batch를 계속 재구성합니다.
  • PagedAttention: OS의 virtual memory처럼 KV cache를 page/block 단위로 관리해 fragmentation을 줄이고 더 많은 요청을 수용합니다.
  • Chunked prefill: 긴 prompt prefill을 chunk로 쪼개 decode 작업과 섞어, 한 요청이 GPU를 독점하지 않게 합니다.
  • Speculative decoding: 작은 draft model이 여러 token 후보를 만들고 큰 model이 검증해 decode 병목을 줄입니다.

제품 관점에서는 평균 tokens/sec보다 사용자가 보는 first-token latency, inter-token latency, p99 latency가 더 중요할 때가 많습니다.

병렬화 조합: 3D에서 N-D로

실제 대규모 학습은 한 가지 병렬화만 쓰지 않습니다. 보통 data × tensor × pipeline을 기본 3D parallelism으로 보고, 여기에 ZeRO/FSDP, sequence parallel, expert parallel을 더합니다.

상황추천되는 조합이유
dense 모델이 한 GPU에 안 들어감tensor + pipeline + ZeRO/FSDPlayer 내부와 layer 묶음, 모델 상태를 동시에 분산
context가 매우 김sequence/context parallel + FlashAttention류 kerneltoken dimension의 activation/KV 부담 완화
MoE 모델expert parallel + data/tensor parallelexpert를 분산하고 active compute를 낮춤
실시간 LLM APIcontinuous batching + KV paging + chunked prefill동시 요청과 tail latency 관리

DeepSeek V4와 연결해서 읽기

  • MoE 스케일링: V4-Pro/V4-Flash처럼 total parameter와 active parameter를 분리해서 말하는 모델은 expert parallelism과 routing 효율이 핵심입니다.
  • 1M context: 긴 context는 position encoding 문제만이 아니라 prefill 병렬화, KV cache placement, sequence parallelism 문제입니다.
  • Hybrid attention: CSA/HCA 같은 구조가 attention/KV 비용을 줄여도, 실제 serving에서는 tensor parallel, KV paging, scheduler가 함께 맞아야 합니다.
  • Agentic coding: repo 전체를 넣는 작업은 prefill이 길고 decode도 길 수 있습니다. throughput보다 작업 단위 latency와 실패 복구 비용을 봐야 합니다.

자주 하는 오해

  • “GPU 수를 늘리면 선형으로 빨라진다”: 통신, bubble, load imbalance 때문에 보통 선형 scaling은 어렵습니다.
  • “tensor parallel만 쓰면 된다”: tensor parallel은 빠른 node 내부 통신에 적합하지만, cluster 전체 확장에는 pipeline/data/expert 조합이 필요합니다.
  • “학습 병렬화와 추론 병렬화는 같다”: 추론은 KV cache, variable-length request, tail latency가 별도 문제입니다.
  • “MoE는 무조건 싸다”: active FLOPs는 줄지만 routing, all-to-all, expert imbalance가 비용을 다시 만들 수 있습니다.
  • “1M context는 모델 스펙이다”: 스펙이기도 하지만, 실제 사용 가능성은 serving 병렬화와 scheduler 품질에 크게 좌우됩니다.

한 줄 요약

LLM parallelism은 “모델을 여러 GPU에 올리는 기술”이 아니라, 학습과 서빙의 병목을 파라미터·activation·sequence·expert·KV cache·request라는 여러 축으로 분해해 배치하는 시스템 설계입니다.

단답 주관식

  1. Tensor parallelism은 Transformer의 어떤 단위를 주로 나누는가?

    Layer 내부의 큰 행렬 곱/weight matrix 차원을 나눈다.

  2. Pipeline parallelism에서 bubble은 무엇인가?

    pipeline 앞뒤 또는 stage 불균형 때문에 일부 device가 일을 못 하고 비는 시간이다.

  3. ZeRO-3/FSDP가 줄이려는 핵심 메모리 낭비는 무엇인가?

    data parallel 복제본마다 parameter, gradient, optimizer state를 모두 들고 있는 낭비다.

  4. LLM serving에서 prefill과 decode의 병목은 어떻게 다른가?

    prefill은 긴 prompt를 한 번에 처리하는 compute-heavy 단계이고, decode는 KV cache를 반복적으로 읽는 memory/bandwidth-heavy 단계인 경우가 많다.

객관식

  1. MoE 모델에서 expert parallelism이 특히 중요한 이유는?

    expert를 device에 분산하고 token을 라우팅해야 하기 때문

  2. Ring Attention과 가장 가까운 병렬화 축은?

    sequence/context parallelism

  3. PagedAttention이 주로 다루는 serving 자원은?

    KV cache memory

  4. 실제 대규모 LLM 학습에서 흔한 접근은?

    data, tensor, pipeline, ZeRO/FSDP, expert 등을 조합
