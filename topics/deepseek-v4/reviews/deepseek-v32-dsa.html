DeepSeek-V3.2 / DeepSeek Sparse Attention

V4의 CSA/HCA hybrid attention을 이해하기 위한 직접 선행. dense attention 비용을 줄이는 DeepSeek 계열 sparse attention 흐름입니다.

원문/자료 보기 리뷰 목록

문제의식

긴 context에서는 새 token 하나를 생성할 때도 과거 모든 token에 attention을 계산하고 KV cache를 유지해야 합니다. context가 수십만~백만 token으로 늘어나면 attention FLOPs와 cache memory가 제품화를 막는 병목이 됩니다.

방법

DeepSeek Sparse Attention은 모든 token을 동일하게 보지 않고, query에 관련 있는 일부 token/block을 선택적으로 attend하는 방향입니다. 핵심은 lightning indexer가 각 query token에 중요한 과거 KV entry를 점수화하고 top-k만 골라 core attention을 수행하는 구조입니다. 먼저 dense warm-up 단계에서 indexer를 dense attention 분포에 KL loss로 정렬하고, 이후 sparse training으로 전체 모델을 sparse pattern에 적응시키는 흐름으로 이해할 수 있습니다.

기여

DeepSeek 계열 모델에서 long-context 효율을 모델 구조 차원에서 해결하려는 중요한 중간 단계입니다. V3.2가 sparse attention의 가능성을 보여줬다면, V4는 CSA/HCA처럼 압축과 희소 선택을 더 강하게 결합한 형태로 확장합니다.

DeepSeek V4와의 연결

V4 발표의 핵심인 CSA/HCA는 “긴 context를 그냥 지원하는 것”이 아니라 “긴 context의 attention과 KV cache 비용을 줄이는 것”입니다. DSA는 이 문제의식의 전 단계로 볼 수 있습니다.

한계와 읽을 때 던질 질문

sparse attention의 리스크는 중요한 먼 문맥을 놓치는 것입니다. 특히 법률/코드/과학 문서처럼 단 하나의 token이나 함수 정의가 중요한 경우 selection 실패가 답변 오류로 이어질 수 있습니다.
