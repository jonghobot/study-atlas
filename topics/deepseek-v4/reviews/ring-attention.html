Ring Attention with Blockwise Transformers for Near-Infinite Context

긴 context를 “한 GPU에 더 많이 올리는 문제”가 아니라 sequence dimension을 분산하는 시스템 문제로 재정의한 논문입니다. DeepSeek V4의 장문맥 주장을 읽을 때, 모델 구조와 serving 병렬화가 어디서 만나는지 보는 기준점이 됩니다.

원문/자료 보기 리뷰 목록

30초 요약

Ring Attention은 long-context Transformer를 위해 sequence를 여러 device에 나눠 올리고, key/value block을 ring topology로 순환시키며 attention을 계산하는 방법입니다. 핵심은 full attention을 포기하지 않으면서 단일 device 메모리 한계를 넘는 것입니다. 계산 중에는 다음 KV block 통신을 미리 진행해 communication과 computation을 overlap합니다.

이 논문을 DeepSeek V4 맥락에서 읽으면, “1M context” 같은 주장이 단순히 RoPE scaling이나 attention 변형만으로 끝나는 문제가 아니라 KV cache 배치, interconnect, scheduler, tail latency까지 포함한 시스템 설계 문제라는 점이 선명해집니다.

문제의식

Transformer의 self-attention은 모든 query token이 모든 key token을 봅니다. 이 구조는 표현력은 강하지만, sequence 길이 L이 커질수록 attention score 계산은 대략 O(L²), KV 저장량은 O(L)로 커집니다. FlashAttention류의 kernel 최적화가 memory I/O를 줄여도, 단일 device에 올려야 하는 token 수가 계속 늘어나면 결국 context window 확장이 막힙니다.

Ring Attention의 핵심 질문은 단순합니다. “모델 파라미터가 아니라 token sequence를 여러 device에 나눠 담으면, approximation 없이 더 긴 context를 처리할 수 있는가?”

핵심 아이디어

Sequence를 block으로 쪼개고, 각 device는 자기 query block을 유지합니다. Key/value block은 device들 사이를 ring 형태로 순환하고, 각 device는 도착한 KV block에 대해 blockwise attention을 계산합니다.

중요한 점은 통신과 계산을 겹친다는 것입니다. 다음 KV block이 ring을 타고 오는 동안 현재 block의 attention을 계산해, device 수에 비례해 sequence 길이를 늘리면서도 추가 overhead를 숨기려 합니다.

왜 approximation이 아닌가

이 접근은 sparse attention처럼 일부 token만 보거나, retrieval처럼 context를 외부에서 고르는 방식이 아닙니다. Blockwise로 나눠 계산하지만 softmax normalization을 streaming 방식으로 유지해 full attention과 같은 결과를 목표로 합니다.

따라서 “긴 문맥을 대충 압축해서 본다”가 아니라 “전체 문맥을 보되, 계산 순서와 데이터 배치를 바꾼다”에 가깝습니다.

논문의 주장

저자들은 Ring Attention이 기존 memory-efficient Transformer가 처리할 수 있는 sequence보다 device 수에 비례해 더 긴 sequence를 학습·추론할 수 있다고 주장합니다. language modeling과 reinforcement learning 실험에서 million-token급 context 가능성을 보입니다.

다만 이 주장은 모델 품질 하나만의 성과라기보다, interconnect bandwidth, pipeline overlap, block 크기, activation checkpointing이 함께 맞아떨어질 때 가능한 시스템 성과로 읽어야 합니다.

작동 방식: query는 고정, KV가 돈다

  1. Sequence partitioning: 전체 token sequence를 여러 block으로 나누고 device마다 일부 block을 맡깁니다.
  2. Local query: 각 device는 자기 query block을 기준으로 attention output을 누적합니다.
  3. Ring communication: key/value block이 이웃 device로 전달됩니다. 한 step이 끝날 때마다 각 device는 다른 구간의 KV를 받습니다.
  4. Blockwise attention: 받은 KV block에 대해 attention partial result를 계산합니다.
  5. Streaming softmax merge: block별 partial score를 numerically stable하게 병합해 전체 attention 결과와 맞춥니다.
  6. Overlap: 현재 block을 계산하는 동안 다음 block 통신을 진행해 통신 시간을 숨깁니다.

직관적으로는 “각 GPU가 자기 질문지를 들고 있고, 참고자료 묶음(KV)이 원탁을 돌며 지나갈 때마다 답안을 조금씩 채우는 방식”입니다.

Streaming softmax는 왜 필요한가

일반적인 attention은 먼저 모든 score QKᵀ를 만든 뒤 softmax를 적용하고, 그 weight로 V를 가중합합니다. 문제는 긴 context에서는 모든 score matrix를 한 번에 메모리에 올릴 수 없다는 점입니다. Ring Attention처럼 KV block을 나눠 받아 계산하려면, “지금 본 block의 softmax 결과”를 나중에 도착할 block과 정확히 합칠 수 있어야 합니다.

Streaming softmax는 이 문제를 해결하기 위해 각 query마다 두 값을 계속 들고 갑니다. 하나는 지금까지 본 score의 최대값 m, 다른 하나는 exp score의 누적합 l입니다. 새 block의 score가 들어오면 새 최대값 m_new를 기준으로 이전 누적값을 다시 스케일링하고, 새 block의 exp 합을 더합니다. 이렇게 하면 block을 순서대로 처리해도 전체 token을 한 번에 softmax한 것과 같은 normalization을 만들 수 있습니다.

직관
새 KV block이 올 때마다 “지금까지 본 후보들 중 최고 점수” 기준을 갱신하고, 이전 가중합을 새 기준에 맞춰 재조정합니다. 그래서 뒤늦게 더 큰 score가 나타나도 앞 block 결과를 버리지 않고 정확히 보정할 수 있습니다.
효과
Attention score 전체를 저장하지 않아도 됩니다. 각 query block은 현재 output 누적값, 최대 score, normalization 누적합만 유지하면 되므로 blockwise/ring 계산과 잘 맞습니다.

이 부분이 빠지면 block별 softmax를 따로 계산한 뒤 평균내는 식이 되어 full attention과 달라집니다. Ring Attention이 “approximation 없이” 긴 context를 다룬다고 말할 수 있는 이유 중 하나가 바로 이 streaming softmax merge입니다.

DeepSeek V4와의 연결

DeepSeek V4 쪽에서 CSA/HCA, KV cache 효율화, long-context scaling을 본다면 Ring Attention은 보완적인 관점입니다. 전자는 한 요청을 더 싸게 처리하는 모델·메모리 구조의 문제에 가깝고, 후자는 context dimension 자체를 여러 device로 펼치는 분산 실행 문제에 가깝습니다.

  • 모델 구조 관점: attention 계산량과 KV cache를 줄이면 같은 hardware에서 더 긴 context 또는 더 많은 동시 요청을 처리할 수 있습니다.
  • 시스템 관점: sequence parallelism을 잘 설계하면 단일 device 메모리 한계를 넘어 context window를 확장할 수 있습니다.
  • serving 관점: 실제 제품에서는 평균 throughput보다 p95/p99 latency, KV 이동 비용, scheduler 복잡도가 더 큰 병목이 될 수 있습니다.
  • 평가 관점: “1M token 입력 가능”과 “1M token 안에서 필요한 정보를 찾아 정확히 추론”은 별개입니다. Ring Attention은 전자를 위한 실행 전략이고, 후자는 모델 학습·평가 문제입니다.

한계와 주의점

  • 통신 의존성: ring 구조는 device 간 bandwidth와 latency에 민감합니다. NVLink/InfiniBand급 환경과 일반 cloud interconnect의 체감 성능은 다를 수 있습니다.
  • 작은 batch·interactive serving: 긴 문서 하나를 빠르게 답해야 하는 agent use case에서는 overlap 효율이 충분히 나오지 않을 수 있습니다.
  • 구현 복잡도: numerically stable streaming softmax, activation checkpointing, attention kernel, distributed runtime이 함께 맞아야 합니다.
  • Straggler 문제: ring의 한 device가 늦어지면 전체 step이 영향을 받습니다. 대규모 serving에서는 placement와 장애 격리가 중요합니다.
  • 품질 문제는 별개: million-token을 “넣을 수 있음”과 그 안에서 정확히 찾고 추론할 수 있음은 다릅니다. Needle-in-a-haystack, long-range dependency, multi-hop QA를 따로 봐야 합니다.

실제로 읽을 때 봐야 할 포인트

  • Block size: block이 너무 작으면 통신 overhead가 커지고, 너무 크면 memory pressure와 overlap 효율 문제가 생깁니다.
  • Interconnect: 논문의 성능 주장은 어떤 device topology에서 측정됐는지 봐야 합니다. GPU 내부/서버 내부/서버 간 통신은 완전히 다른 문제입니다.
  • Training vs inference: 학습에서는 activation과 gradient까지 고려해야 하고, 추론에서는 prefill과 decode 단계의 병목이 다릅니다.
  • Prefill vs decode: 긴 prompt를 한 번에 읽는 prefill에는 sequence parallelism이 잘 맞을 수 있지만, token-by-token decode는 KV cache 접근 패턴이 달라집니다.
  • End-to-end metric: TFLOPs나 token length보다 실제 사용자가 보는 latency, cost/request, 정확도, 실패율을 함께 봐야 합니다.

한 줄 요약

Ring Attention은 long context를 가능하게 하는 “마법 attention”이라기보다, full attention을 유지한 채 sequence를 device 축으로 분산하는 실행 전략입니다. DeepSeek V4 같은 장문맥 모델을 평가할 때는 모델 구조의 효율성과 분산 serving의 현실성을 따로 놓고 봐야 합니다.
