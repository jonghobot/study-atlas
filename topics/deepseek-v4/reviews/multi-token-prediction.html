Multi-Token Prediction: 한 번에 다음 여러 토큰을 예측하는 학습/추론 전략

MTP는 “다음 1개 토큰”만 맞히던 언어모델 학습을 “다음 여러 개 토큰”까지 넓히는 방법입니다. long-context 길이를 직접 늘리지는 않지만, 모델이 더 먼 미래 구조를 보게 만들고 speculative decoding의 draft 생성기로도 이어져 학습 효율·코딩 성능·생성 속도와 연결됩니다.

관련 논문 보기 리뷰 목록

문제의식

표준 autoregressive LM은 위치 t에서 오직 x(t+1) 하나만 예측합니다. 이 목표는 단순하고 안정적이지만, 모델은 매번 “바로 다음 단어”에만 직접적인 손실을 받습니다. 코드처럼 몇 토큰 뒤의 괄호, 들여쓰기, 변수 재사용, 문장 완결성이 중요한 영역에서는 더 긴 패턴을 학습하는 신호가 약할 수 있습니다.

핵심 아이디어

MTP는 같은 hidden state 또는 추가 MTP block 위에 여러 prediction head를 붙여 x(t+1), x(t+2), ... x(t+n)을 함께 예측합니다. 즉 “지금까지의 문맥을 보고 다음 한 글자만 맞혀라”가 아니라 “다음 몇 걸음의 전개까지 동시에 예상해봐라”라고 학습시키는 보조 objective입니다.

기여

Gloeckle et al.(2024)은 공유 trunk 위에 독립 output head들을 두는 방식으로 multi-token prediction을 auxiliary task로 사용했습니다. 논문은 학습 시간 overhead 없이 sample efficiency와 downstream 성능이 좋아졌고, 특히 coding benchmark에서 이득이 컸다고 보고합니다. 13B 모델 기준 HumanEval은 12%, MBPP는 17% 더 많은 문제를 풀었고, 4-token prediction 모델은 inference에서 최대 3배 빠른 속도도 보였습니다.

DeepSeek V4와의 연결

DeepSeek V4의 핵심 축은 1M context와 hybrid attention이지만, 실제 agentic coding에서는 긴 context만큼 생성 속도도 중요합니다. DeepSeek V3/V4 계열에서 언급되는 MTP module은 주 모델 뒤에 붙어 추가 미래 토큰을 예측하고, 이 출력을 speculative decoding의 초안 토큰으로 활용할 수 있는 구조적 근거가 됩니다.

개념을 수식 없이 풀어보기

문장 “오늘 저녁에는 치킨을”까지 읽은 모델을 생각해봅시다. 일반 next-token prediction은 다음 토큰 “먹” 또는 “시켜” 같은 후보 하나만 맞히도록 학습합니다. MTP는 여기서 멈추지 않고, 같은 위치에서 “먹 / 고 / 싶 / 다”처럼 앞으로 이어질 여러 토큰을 각각 예측하게 합니다.

학습 목표모델이 직접 예측하는 것배우는 신호
Next-token prediction다음 1개 토큰바로 다음 단어 선택
Multi-token prediction다음 2~4개 이상의 토큰짧은 미래 전개, 구문 완결성, 반복/유도 패턴

중요한 점은 MTP가 “한 번에 여러 토큰을 확정 출력한다”는 뜻만은 아니라는 것입니다. 학습 단계에서는 여러 미래 토큰을 맞히는 보조 손실이고, 추론 단계에서는 그 예측을 초안 후보로 써서 빠르게 검증할 수 있습니다.

작동 방식: 학습과 추론을 나눠 보기

1) 학습 시

본문 모델은 기존처럼 prefix를 읽고 hidden state를 만듭니다. 그 위에 여러 head 또는 MTP module을 붙여 depth 1 head는 다음 토큰, depth 2 head는 다다음 토큰, depth 3 head는 세 번째 미래 토큰을 예측합니다. 손실은 각 depth의 cross-entropy를 합치거나 평균내어 계산합니다.

2) 추론 시

기본 decoding은 여전히 token-by-token으로 할 수 있습니다. 다만 MTP head가 “앞으로 나올 법한 여러 토큰”을 draft로 제안하면, 주 모델이 이를 검증해 받아들일 수 있습니다. 여러 draft가 한 번에 accept되면 같은 품질을 유지하면서 wall-clock latency가 줄어듭니다.

실례 1: 코드 생성에서 왜 효과가 큰가

코드는 자연어보다 다음 몇 토큰의 구조가 강하게 묶여 있습니다. 예를 들어 Python에서 모델이 for i in range(n):까지 생성했다면, 다음은 줄바꿈, 들여쓰기, 변수 사용, 누산 또는 조건문이 이어질 가능성이 큽니다.

문맥일반 next-tokenMTP가 함께 보게 되는 미래
for i in range(n):줄바꿈줄바꿈 + 들여쓰기 + total += arr[i] 같은 들여쓰기·변수 재사용 패턴
if x is None:줄바꿈줄바꿈 + 들여쓰기 + return None 같은 블록 완결 패턴
def add(a, b):줄바꿈줄바꿈 + 들여쓰기 + return a + b 같은 함수 본문 전개

이런 영역에서는 “다음 1개 토큰”만 맞히는 것보다 “몇 토큰 뒤까지 자연스럽게 이어지는가”가 품질을 좌우합니다. 그래서 MTP는 HumanEval, MBPP 같은 생성형 코딩 benchmark에서 특히 유리하게 나타납니다.

실례 2: speculative decoding과 연결

Speculative decoding은 작은 draft 모델이 여러 토큰을 먼저 제안하고, 큰 target 모델이 이를 빠르게 검증하는 방식입니다. MTP는 별도 작은 모델 대신 같은 모델 내부의 MTP head/module이 draft를 낼 수 있다는 점에서 self-speculative decoding과 잘 맞습니다.

  1. MTP module이 다음 후보 3~4개 토큰을 빠르게 제안합니다.
  2. 주 모델이 각 후보가 자기 분포에서도 타당한지 검증합니다.
  3. 연속 후보가 맞으면 여러 토큰을 한 번에 accept합니다.
  4. 틀린 지점이 나오면 그 지점부터 다시 일반 decoding으로 복구합니다.

accept rate가 높을수록 속도 이득이 큽니다. 반대로 draft가 자주 틀리면 검증 비용 때문에 이득이 작아집니다.

한계와 읽을 때 던질 질문

MTP가 항상 품질을 높이는 것은 아닙니다. 여러 미래 토큰을 예측하는 objective가 주 next-token objective와 충돌할 수 있고, depth가 깊어질수록 미래 토큰의 불확실성이 커집니다. speculative decoding으로 이득을 보려면 draft token accept rate가 충분히 높아야 하며, batching·KV cache·GPU kernel 구현과도 맞물립니다. 또한 MTP는 긴 context에서 중요한 병목인 KV cache memory 자체를 직접 줄이는 기술은 아닙니다.

이 페이지에서의 위치

MTP는 long-context extension보다는 decoding efficiency / training objective 축에 넣는 게 맞습니다. DeepSeekMoE가 모델 용량 대비 계산량, DSA/CSA/HCA가 긴 context attention 비용, KV cache compression이 memory 병목을 다룬다면, MTP는 생성 단계의 token throughput과 미래 token 학습 신호를 다룹니다.

이해 확인 퀴즈

단답 주관식

  1. Multi-Token Prediction은 기존 next-token prediction과 무엇이 다른가?

    다음 한 토큰뿐 아니라 여러 미래 토큰을 함께 예측하도록 학습 신호를 준다.

  2. MTP가 decoding throughput과 연결되는 이유는?

    여러 토큰 예측이 speculative decoding이나 병렬적 후보 생성에 활용될 수 있기 때문이다.

  3. MTP가 long-context 자체를 직접 해결하지는 않는 이유는?

    context 저장·attention 비용보다 생성 단계의 예측/decoding 효율에 더 가까운 기법이기 때문이다.

  4. MTP의 품질 리스크는 무엇인가?

    멀리 있는 미래 토큰 예측이 부정확하면 학습 신호나 speculative accept rate가 기대보다 낮을 수 있다.

객관식

  1. MTP의 핵심은?

    여러 미래 토큰 예측

  2. MTP가 특히 연결되는 추론 최적화는?

    speculative decoding

  3. MTP가 주로 겨냥하는 병목은?

    생성/decoding 효율

  4. MTP를 평가할 때 볼 만한 지표는?

    accept rate와 latency 개선
