1. 학습 병렬화 대표 자료

2018
pipeline

GPipe: Efficient Training of Giant Neural Networks using Pipeline Parallelism

GPipe의 문제의식은 모델을 키우면 품질은 좋아지지만, 한 accelerator의 메모리에 들어가지 않는 순간 연구자가 모델 구조를 바꾸거나 특수 병렬화 코드를 직접 짜야 한다는 데 있습니다. GPipe는 신경망이 층의 순서열이라는 점에 기대어, 연속된 layer 구간을 여러 장치에 나누고 mini-batch를 micro-batch로 쪼개 pipeline처럼 흘려보냅니다. 전방향 계산이 앞 stage에서 시작되면 다음 micro-batch가 뒤따르고, 역전파도 같은 구조로 이어져 장치 idle을 줄입니다.

핵심 병렬 축은 layer-wise pipeline model parallelism입니다. 데이터 병렬처럼 각 장치가 모델 전체를 복제하지 않기 때문에 거대 모델을 수용하기 좋지만, stage 간 activation 전달과 pipeline bubble이라는 비용이 생깁니다. stage별 계산량이 균형을 이루지 않으면 느린 stage가 전체 throughput을 결정하고, micro-batch 수가 적으면 bubble이 커지며, 많으면 activation 저장·재계산 계획이 복잡해집니다. 지금 중요한 이유는 오늘날 LLM 학습의 기본 조합인 pipeline/tensor/data parallel 중 pipeline 축의 표준 어휘를 만들었기 때문입니다.

2019
tensor

Megatron-LM: Training Multi-Billion Parameter Language Models Using Model Parallelism

Megatron-LM은 Transformer layer 자체가 너무 커져 단일 GPU 메모리와 연산량이 병목이 되는 문제를 다룹니다. GPipe가 layer 단위로 모델을 자른다면, Megatron-LM은 한 layer 내부의 행렬곱을 여러 GPU에 나눕니다. self-attention의 QKV projection, output projection, MLP의 선형층을 column/row parallel 방식으로 쪼개고, 필요한 지점에 all-reduce 같은 collective communication을 삽입합니다.

병렬 축은 hidden dimension, attention head, MLP intermediate dimension입니다. 장점은 큰 layer를 여러 GPU에 담을 수 있고, NVLink/NVSwitch처럼 빠른 intra-node 연결에서는 통신비를 감당할 수 있다는 점입니다. 반대로 tensor parallel group이 느린 cross-node 네트워크로 퍼지면 빈번한 collective가 치명적입니다. degree를 너무 키우면 per-GPU matmul이 작아져 kernel efficiency도 떨어집니다. 현재 dense LLM 학습 스택 대부분이 tensor parallelism을 기본 축으로 삼는다는 점에서, 이 논문은 여전히 가장 중요한 출발점입니다.

2019
memory

ZeRO: Memory Optimizations Toward Training Trillion Parameter Models

ZeRO는 데이터 병렬 학습의 숨은 낭비에서 출발합니다. 데이터 병렬은 구현이 쉽지만 모든 GPU가 parameter, gradient, optimizer state를 똑같이 복제합니다. Adam을 쓰면 optimizer state가 parameter보다 더 큰 메모리를 차지하므로, 모델이 커질수록 복제 비용이 먼저 한계가 됩니다. ZeRO는 이 중복을 제거하기 위해 optimizer state, gradient, parameter를 data parallel group 안에서 shard합니다.

Stage 1은 optimizer state, Stage 2는 gradient, Stage 3는 parameter까지 나눕니다. 각 장치는 자신이 맡은 조각만 보관하고, 계산 시 필요한 값은 reduce-scatter와 all-gather로 맞춥니다. trade-off는 명확합니다. 메모리는 줄지만 통신과 runtime 복잡도가 늘어납니다. bucket 크기, overlap, checkpointing, mixed precision, CPU/NVMe offload와 결합될 때 성능이 크게 달라집니다. 오늘날 DeepSpeed ZeRO와 PyTorch FSDP가 모두 이 문제의식 위에 있으며, LLM 학습에서 메모리 최적화가 병렬화의 핵심 축이라는 사실을 대중화했습니다.

2020
MoE

GShard: Scaling Giant Models with Conditional Computation and Automatic Sharding

GShard는 dense 모델이 parameter를 키우면 계산량도 함께 커지는 문제와, 수천 TPU에 모델을 나누는 sharding 계획이 너무 어렵다는 문제를 함께 겨냥합니다. Transformer에 sparsely-gated Mixture-of-Experts를 넣어 token마다 일부 expert만 활성화하고, XLA compiler 확장과 annotation API로 sharding을 자동화합니다. 전체 parameter 수는 크게 키우되 token당 계산량은 제한하는 방식입니다.

핵심 병렬 축은 expert parallelism과 compiler-assisted sharding입니다. MoE layer에서 token을 expert로 라우팅하고 expert parameter를 장치들에 분산시키지만, dispatch/combine 과정에서 all-to-all 통신이 발생합니다. expert load balancing이 실패하면 일부 장치만 과부하되고 나머지는 놀게 됩니다. GShard는 이후 Switch Transformer와 대형 sparse model 흐름의 기반이 되었고, “모든 token에 모든 parameter를 쓰지 않아도 된다”는 관점을 대규모 시스템에서 증명했습니다.

2021
MoE

Switch Transformers

Switch Transformer는 MoE가 강력하지만 복잡하고 불안정하다는 문제에서 출발합니다. 기존 MoE는 token을 여러 expert에 보낼 수 있어 표현력은 좋지만 routing, communication, load balancing, precision 안정성 문제가 큽니다. Switch의 핵심은 radical simplification입니다. 각 token을 top-1 expert 하나에만 보내 통신과 combine 비용을 줄이고, 같은 compute budget에서 parameter 수를 trillion scale까지 늘립니다.

trade-off는 단순성에 따른 capacity와 안정성입니다. top-1 routing은 빠르지만 expert 선택이 편향되면 token overflow가 생깁니다. capacity factor, auxiliary load-balancing loss, router z-loss, selective precision 같은 장치가 필요합니다. Sparse 모델은 parameter 수가 크다고 dense 모델처럼 자동으로 좋아지는 것이 아니며, serving과 fine-tuning에서도 expert 배치와 통신 구조가 중요합니다. 오늘날 대형 MoE LLM의 기본 직관을 가장 단순한 형태로 보여준 자료입니다.

2021
3D

Efficient Large-Scale Training with Megatron-LM

이 논문은 단일 병렬화 기법만으로는 trillion-scale 학습이 어렵다는 점을 보여줍니다. tensor parallelism은 layer 내부 matmul을 나누지만 cross-node 통신에 취약하고, pipeline parallelism은 layer를 나누지만 bubble과 imbalance가 있으며, data parallelism은 throughput을 늘리지만 모델 상태 복제가 문제입니다. 논문은 data × tensor × pipeline을 조합하는 3D parallelism을 체계적으로 분석합니다.

실용 recipe는 topology-aware합니다. tensor parallel group은 빠른 intra-node 연결 안에 두고, pipeline은 node 간 stage로 확장하며, 남는 축을 data parallel로 사용합니다. interleaved pipeline schedule은 physical device가 여러 virtual pipeline stage를 맡게 해 bubble을 줄입니다. 하지만 tensor degree, pipeline depth, micro-batch 수, global batch size, activation recomputation, 네트워크 토폴로지가 모두 얽혀 최적점이 달라집니다. 오늘날 대형 LLM 학습에서 중요한 것은 개별 기법보다 병렬 축들의 조합이라는 점을 정리한 기준점입니다.

2022
compiler

Alpa: Automating Inter- and Intra-Operator Parallelism

Alpa는 병렬화 계획을 사람이 계속 손으로 짤 수 있는가라는 질문에서 출발합니다. 실제 모델과 클러스터가 복잡해질수록 어떤 operator를 shard하고, 어떤 layer를 pipeline stage에 놓고, 어떤 device mesh를 쓸지 결정하는 일은 어려워집니다. Alpa는 병렬화를 inter-operator parallelism과 intra-operator parallelism으로 나눕니다. 전자는 computation graph를 pipeline stage로 나누는 문제이고, 후자는 각 operator 내부 tensor sharding을 정하는 문제입니다.

핵심은 병렬화 config를 compiler/search 문제로 바꾸는 것입니다. 수작업으로 TP/PP/DP를 고르는 대신, cost model과 graph partitioning을 통해 장치 mesh에 맞는 계획을 찾습니다. trade-off는 자동화가 항상 최적 실무 성능을 보장하지는 않으며, 모델 패턴과 runtime 제약에 민감하다는 점입니다. 그래도 Alpa는 LLM 병렬화가 점점 사람이 외우는 recipe에서 시스템이 탐색하는 영역으로 이동한다는 방향을 잘 보여줍니다.

2. Long-context / sequence parallelism

2023
sequence

DeepSpeed Ulysses

DeepSpeed Ulysses는 장문맥 LLM 학습에서 기존 데이터/텐서/파이프라인 병렬화가 잘 다루지 못했던 sequence length 축을 직접 분할합니다. 입력 시퀀스를 여러 GPU에 나누고, attention 계산에 필요한 Q/K/V 재배치를 all-to-all 통신으로 수행합니다. 긴 문맥으로 갈수록 activation과 attention 메모리가 커지기 때문에 token dimension 자체를 분산해야 한다는 점을 명확히 보여줍니다.

장점은 sequence 길이와 장치 수를 함께 늘릴 수 있고 Megatron류 tensor parallelism과 결합 가능하다는 점입니다. 단점은 all-to-all 성능과 네트워크 topology에 매우 민감하다는 것입니다. 특히 cluster 규모가 커지면 communication scheduling과 overlap이 전체 효율을 좌우합니다. Ulysses는 long-context training에서 “attention을 sequence 축에서 어떻게 재분배할 것인가”라는 질문을 제기한 대표 자료입니다.

2023
context

Ring Attention

Ring Attention은 긴 시퀀스를 block 단위로 나누고 각 장치가 ring 형태로 K/V block을 주고받으며 blockwise self-attention을 계산합니다. Ulysses가 all-to-all 재배치에 가깝다면, Ring Attention은 통신을 단계적으로 순환시키고 K/V block 전송을 계산과 overlap하는 데 초점을 둡니다. streaming softmax를 사용해 block별 attention 결과를 full attention과 일치하도록 병합합니다.

이 접근은 approximation이 아니라 실행 순서와 데이터 배치를 바꾸는 방식입니다. 장점은 device 수만큼 긴 context를 다룰 수 있고 all-to-all보다 네트워크 요구가 완만할 수 있다는 점입니다. 단점은 ring step이 늘어나면 latency와 scheduling 복잡도가 커진다는 점입니다. 백만 토큰 context 같은 주장을 볼 때, 모델 구조뿐 아니라 interconnect, block size, overlap, kernel efficiency가 모두 중요하다는 사실을 보여줍니다.

2024
USP

USP: Unified Sequence Parallelism

USP는 DeepSpeed Ulysses와 Ring Attention을 비교·통합하는 관점에서 제안된 unified sequence parallelism입니다. 특정 Transformer 구조나 네트워크 topology에 따라 Ulysses식 all-to-all과 Ring식 순환 통신의 효율이 달라지므로, 두 계열을 하나의 프레임으로 해석하고 더 강건한 병렬 attention 구성을 제공하려 합니다.

논문은 data/tensor/ZeRO/pipeline parallelism과 SP의 메모리·통신 비용을 비교하고, SP를 포함한 hybrid 4D parallelism 설계 지침을 제시합니다. Llama3-8B를 208K sequence로 학습한 실험은 이 접근이 연구용 아이디어를 넘어 실용 영역에 가까워지고 있음을 보여줍니다. 개별 알고리즘보다 “SP를 기존 병렬화 스택에 어떻게 넣을 것인가”에 초점이 있습니다.

2024
4D

4D Parallelism and Memory Consumption Estimator

이 논문은 DP, TP, PP, CP를 조합하는 4D 병렬화의 메모리 사용량을 정량적으로 예측하는 데 초점을 맞춥니다. Llama 계열 구조를 대상으로 parameter, gradient, optimizer state, activation 메모리 공식을 제시하고, A100/H100 실험으로 검증합니다. temporary buffer와 memory fragmentation처럼 실제 OOM에 영향을 주지만 자주 무시되는 요인을 포함한 점이 실용적입니다.

장문맥 학습에서는 CP/SP가 activation과 attention 메모리를 줄일 수 있지만, TP/PP/DP와 섞이는 순간 최적 조합이 직관적이지 않습니다. 이 자료는 새로운 attention 알고리즘이라기보다 “어떤 병렬화 조합이 실제 GPU 메모리에 들어가는가”를 빠르게 판단하는 engineering tool에 가깝습니다. 대규모 실험 전에 config 후보를 줄이는 용도로 가치가 큽니다.

3. Serving / runtime 병렬화

2023
KV

PagedAttention / vLLM

LLM serving의 병목은 FLOPs만이 아니라 요청마다 길이가 다르고 생성 중 계속 커지는 KV cache입니다. 기존 시스템은 요청 단위로 연속 메모리를 크게 잡거나 최대 길이를 보수적으로 예약해 단편화와 중복 저장이 컸습니다. PagedAttention은 OS의 가상 메모리처럼 KV cache를 고정 크기 block으로 나누고, 논리 token 위치를 물리 block에 매핑합니다.

vLLM은 이 구조 위에서 요청이 실제 token을 생성할 때만 block을 할당하고, copy-on-write로 여러 후보가 같은 prefix KV를 공유하게 합니다. 모델 병렬 자체보다 serving-level parallelism을 가능하게 한 것이 대표성입니다. 메모리 낭비를 줄이면 continuous batching에서 더 많은 요청을 동시에 decode할 수 있습니다. 긴 context, RAG, agent, multi-sample decoding이 보편화된 지금 KV를 얼마나 낭비 없이 담고 공유하느냐가 처리량의 출발점이 되었습니다.

2023
scheduler

SARATHI

SARATHI는 prefill과 decode의 성격이 다르다는 점에서 출발합니다. prefill은 prompt 전체를 병렬 처리하므로 compute-heavy이고, decode는 요청마다 한 token씩만 생성해 arithmetic intensity가 낮습니다. 긴 prefill을 여러 동일한 chunk로 쪼개고, 각 batch에 하나의 prefill chunk와 가능한 많은 decode를 넣는 decode-maximal batching을 제안합니다.

prefill chunk가 GPU compute를 채우는 동안 decode 요청들이 piggyback하므로 decode-only batch보다 추가 비용이 작아집니다. 핵심은 phase-aware micro-batch scheduling입니다. chunk 크기가 작으면 scheduling 유연성은 커지지만 overhead가 늘고, 크면 GPU 효율은 좋지만 piggyback 기회가 줄어듭니다. 이후 chunked prefill과 prefill/decode 혼합 스케줄링은 긴 prompt와 실시간 decode를 함께 처리하는 기본 기법이 되었습니다.

2023
disagg

Splitwise

Splitwise는 prefill은 compute-intensive이고 decode는 memory-intensive인데 대부분의 서버가 두 단계를 같은 GPU 풀에서 실행한다는 문제를 지적합니다. prompt computation과 token generation을 서로 다른 machine에 배치하고, prefill은 compute가 강한 GPU에서, decode는 더 낮은 비용·전력의 장비나 memory bandwidth 중심 장비에서 실행합니다.

장점은 phase별 최적 장비 선택과 독립 scaling입니다. 비용은 prefill 후 생성된 KV cache를 네트워크로 decode machine에 옮겨야 한다는 점입니다. prompt가 길수록 prefill compute도 크고 KV도 커지므로, 전송 비용과 compute 절감 사이 균형이 중요합니다. Splitwise는 “모든 phase를 최고 GPU에서 돌릴 필요가 없다”는 관점을 제시했고, 이후 disaggregated serving 흐름의 출발점이 되었습니다.

2023/24
runtime

SGLang

SGLang은 LLM application이 단일 completion이 아니라 agent loop, tool call, self-consistency, tree-of-thought, JSON constrained decoding처럼 여러 generation call과 control flow를 포함한다는 점에서 출발합니다. frontend language와 runtime을 함께 제안해 generation, fork/join, parallelism control, structured output을 프로그램 수준에서 표현하게 합니다.

대표 최적화인 RadixAttention은 prompt prefix를 radix tree로 관리해 여러 요청·분기 간 공통 KV cache를 재사용합니다. SGLang의 병렬성은 model parallelism보다 program-level parallelism입니다. runtime이 branch generation과 multi-call workflow를 알고 있으므로 어떤 호출을 병렬로 보내고 어떤 prefix를 공유할지 스케줄링할 수 있습니다. LLM serving이 단순 API endpoint에서 application runtime으로 이동하고 있음을 보여주는 자료입니다.

2024
disagg

DistServe

DistServe는 prefill과 decode를 같은 GPU에서 colocate하면 TTFT와 TPOT 최적화가 서로 간섭한다는 점을 다룹니다. prefill을 많이 넣으면 첫 token은 빨라질 수 있지만 decode가 밀리고, decode를 우선하면 긴 prompt의 첫 token이 늦어집니다. DistServe는 prefill과 decoding을 다른 GPU instance에 분리하고, 각 phase에 별도 resource allocation과 parallelism strategy를 적용합니다.

Splitwise와 같은 큰 방향을 공유하지만, DistServe는 SLO와 goodput 관점이 강합니다. 요청을 처리할 수 있는 최대 rate를 TTFT/TPOT 제약 아래에서 최적화하려 하며, KV transfer 비용을 줄이기 위해 cluster bandwidth에 따라 placement를 정합니다. 핵심 trade-off는 interference 감소와 KV 이동 비용입니다. 2024년 이후 disaggregated serving을 평가할 때 기준점으로 자주 언급되는 자료입니다.

2025
kernel

FlashInfer

FlashInfer는 LLM serving에서 attention pattern과 KV cache layout이 점점 다양해지는 문제를 다룹니다. Paged KV, sparse attention, shared prefix, speculative decoding, multi-query/grouped-query attention 등 workload마다 필요한 kernel 형태가 다르지만, 고성능 구현을 매번 따로 만들기 어렵습니다. FlashInfer는 serving용 customizable attention engine으로, 다양한 KV layout과 attention variant를 효율적으로 지원하는 것을 목표로 합니다.

이 자료의 병렬화 축은 scheduler라기보다 kernel/runtime layer입니다. 좋은 scheduler가 batch를 만들더라도 attention kernel이 KV layout을 비효율적으로 읽으면 성능이 무너집니다. FlashInfer는 block-sparse format과 composable format을 통해 memory access를 최적화하고 redundancy를 줄이려 합니다. 2025년 이후 serving stack에서 “KV 관리 + scheduler + attention engine”이 함께 맞아야 한다는 점을 잘 보여줍니다.

2025
KV schedule

Online Scheduling for LLM Inference with KV Cache Constraints

이 논문은 LLM inference scheduling을 KV cache memory 제약이 있는 online scheduling 문제로 모델링합니다. 생성은 token-by-token으로 진행되고 요청마다 output 길이를 사전에 정확히 알기 어렵기 때문에, scheduler는 latency를 줄이면서도 GPU memory budget을 넘지 않아야 합니다. 기존 serving system은 heuristic에 의존하는 경우가 많지만, 이 자료는 KV cache 제약을 수학적으로 명시합니다.

핵심은 batching과 scheduling을 memory-aware하게 설계하는 것입니다. 긴 요청이 KV를 계속 점유하면 새 요청을 받기 어렵고, 짧은 요청을 과도하게 기다리게 하면 latency가 나빠집니다. 이 논문은 vLLM류 시스템 이후의 쟁점이 단순 memory allocator를 넘어 online scheduling 이론과 연결된다는 점을 보여줍니다. 2026년으로 갈수록 KV-aware scheduler 연구가 더 활발해지는 흐름의 전조입니다.

4. 2026 최신 자료

2026 항목은 arXiv preprint 중심입니다. 아직 peer review 여부가 불확실한 자료가 많으므로, “현재 연구 방향”을 파악하는 용도로 보는 것이 안전합니다.

2026
compiler SP

AutoSP: Compiler-Based Sequence Parallelism

AutoSP는 장문맥 학습 최적화를 사용자가 직접 training stack에 삽입해야 하는 문제를 compiler 기반 자동화로 풀려는 자료입니다. 기존 라이브러리는 ZeRO/FSDP, tensor, pipeline 병렬처럼 parameter가 큰 모델을 위한 추상화는 풍부하지만, 수십만 token 문맥을 위한 sequence parallelism과 activation checkpointing 조합은 여전히 수작업 의존도가 높습니다.

AutoSP는 모델 graph를 컴파일해 자동 sequence parallelism과 long-context-aware activation checkpointing을 적용합니다. 핵심 방향은 Ulysses/Ring/USP류 아이디어를 연구용 kernel에서 framework/compiler 최적화로 끌어올리는 것입니다. 실제 도입 시에는 지원되는 모델 pattern, graph capture 제약, 기존 분산 학습 프레임워크와의 호환성이 중요합니다.

2026
PD energy

Revisiting Disaggregated LLM Serving

이 자료는 prefill과 decode를 별도 GPU pool로 나누는 disaggregated serving이 항상 이득이라는 통념을 재검토합니다. PD disaggregation은 prefill의 compute-bound 특성과 decode의 memory-bound 특성을 분리해 TTFT와 TPOT를 각각 최적화할 수 있지만, prefill 완료 후 KV cache를 decode GPU로 옮기는 비용이 필수로 발생합니다.

논문은 여러 KV transfer medium과 colocated baseline을 비교하고, DVFS 기반 profiling으로 성능-에너지 Pareto frontier를 분석합니다. 결론은 신중합니다. disaggregation의 성능 이득은 request load와 KV transfer 경로에 의존하며, stage-wise frequency scaling이 에너지 절감으로 곧장 이어지지는 않습니다. 즉 PD 분리는 유행하는 architecture지만, KV 이동 비용·부하 패턴·에너지까지 포함해 검증해야 합니다.

2026
KV theory

Competitive Non-Clairvoyant KV-Cache Scheduling

이 논문은 LLM inference scheduling을 KV cache 메모리 제약이 있는 online/non-clairvoyant 문제로 모델링합니다. 생성 길이를 미리 알 수 없기 때문에 요청의 memory 사용량은 decode token이 늘어날수록 선형 증가하고, scheduler는 latency를 줄이면서도 hard memory budget을 넘지 않아야 합니다.

기존 연구가 응답 길이를 안다고 가정하거나 ML 예측에 의존했다면, 이 자료는 사전 지식 없이도 worst-case 보장을 갖는 algorithmic approach를 제안합니다. 실무 시스템에 바로 넣기에는 restart 비용과 사용자 경험 문제가 남지만, KV cache scheduling을 이론적으로 엄밀하게 다룬다는 점에서 중요합니다. 2026년의 큰 흐름인 “serving heuristic을 수학적 scheduling 문제로 재정의”하는 방향을 보여줍니다.

2026
multi-turn

Not All Prefills Are Equal: PPD Disaggregation

PPD Disaggregation은 multi-turn chatbot/agent serving에서 기존 PD disaggregation의 비효율을 지적합니다. 일반적인 PD 구조에서는 매 turn마다 새 prompt와 이전 응답을 다시 prefill하고, prefill node와 decode node 사이에 KV transfer가 반복되어 bandwidth 병목과 TTFT 악화를 만듭니다. 핵심 관찰은 Turn 2 이후에는 기존 KV를 재사용하며 새 입력만 처리하는 append-prefill이 가능하다는 점입니다.

PPD는 일부 append-prefill을 decode node에서 local 처리하도록 동적으로 라우팅합니다. 모든 prefill이 같은 비용을 유발하지 않으므로, full prefill과 append-prefill을 구분해야 한다는 주장입니다. multi-turn agent workload가 커질수록 이 관점은 중요합니다. 단일 요청 benchmark보다 실제 대화/agent 세션의 KV 재사용과 placement가 serving architecture의 핵심이 되고 있음을 보여줍니다.

2026
cross-DC

Prefill-as-a-Service

Prefill-as-a-Service는 PD disaggregation의 범위를 단일 고대역폭 cluster에서 cross-datacenter 수준으로 확장할 수 있는지 탐색합니다. 기존 dense attention 모델에서는 prefill이 거대한 KV cache traffic을 만들기 때문에 prefill과 decode가 같은 RDMA급 네트워크 도메인에 묶이는 경우가 많습니다. 그러나 hybrid attention 등으로 KV cache 크기가 줄어들면 원격 prefill cluster 구성이 가능해질 수 있습니다.

자료는 selective offloading, bandwidth-aware scheduling, cache-aware placement를 결합합니다. 주장은 흥미롭지만 내부 모델 기반 case study 성격이 강하므로 일반화 검증은 더 필요합니다. 그래도 중요한 점은 serving parallelism의 단위가 GPU 내부나 단일 cluster를 넘어 datacenter placement 문제로 확장되고 있다는 것입니다.

2026
SSD KV

Tutti: SSD-Backed KV Cache for Long-Context Serving

Tutti는 long-context serving에서 KV cache footprint가 GPU HBM과 CPU DRAM을 넘어 NVMe SSD로 offload되는 상황을 다룹니다. SSD-backed KV는 용량 문제를 완화하지만, cache restore의 I/O 성능이 낮으면 TTFT와 decode latency를 크게 악화시킵니다. 이 자료는 “KV cache를 어디까지 계층화할 수 있는가”라는 practical serving 문제를 전면에 올립니다.

핵심 방향은 prefix caching과 storage hierarchy를 함께 설계하는 것입니다. 긴 context가 일반화될수록 HBM 안에 모든 KV를 유지하는 방식은 비용이 높고, CPU/SSD/offload를 섞는 계층형 cache가 중요해집니다. 단점은 I/O jitter와 tail latency입니다. 2026년 long-context serving에서 memory hierarchy가 GPU scheduling만큼 중요한 축이 되고 있음을 보여줍니다.

2026
KV transfer

SplitZip: Lossless KV Compression for Disaggregated Serving

SplitZip은 disaggregated serving에서 prefill worker가 만든 KV cache를 decode worker로 보내야 하는 비용에 집중합니다. PD disaggregation은 compute-bound prefill과 memory-bound decode를 나눠 load balance를 개선하지만, KV transfer가 병목이 되면 이득이 사라질 수 있습니다. SplitZip은 이 KV handoff를 빠르고 lossless하게 압축하려는 접근입니다.

이 자료가 중요한 이유는 2024–2025년의 disaggregation 논의가 “분리하면 좋다”에서 2026년에는 “분리 후 KV를 어떻게 싸게 옮길 것인가”로 이동했음을 보여주기 때문입니다. 압축률, 압축/해제 latency, 네트워크 bandwidth, decode 시작 지연 사이의 trade-off가 핵심입니다. 품질 손실이 없는 lossless 방향은 안전하지만, 실제 성능은 workload와 KV 구조에 크게 의존할 가능성이 큽니다.

2026
SLO

Taming Request Imbalance: SLO-Aware Scheduling for Disaggregated LLM Inference

이 자료는 production serving에서 요청 길이가 long-tail distribution을 보이며, 이로 인해 prefill side의 head-of-line blocking과 decode side underutilization이 생기는 문제를 다룹니다. disaggregated LLM inference는 phase를 나누지만, request imbalance를 제대로 다루지 않으면 SLO를 만족하기 어렵습니다.

핵심은 SLO-aware scheduling입니다. 평균 throughput보다 사용자에게 중요한 TTFT/TPOT, p95/p99 latency를 직접 목표로 삼아 prefill과 decode resource를 배치해야 합니다. 2026년 serving 연구의 특징은 단순 tokens/sec보다 “SLO를 만족하는 goodput”을 중시한다는 점입니다. 실제 서비스에서는 이 방향이 더 현실적입니다.

5. 다음 확장 제안

  • vLLM/PagedAttention 상세 페이지: block table, copy-on-write, continuous batching을 그림과 예제로 풀면 가장 실용적입니다.
  • PD Disaggregation 비교 페이지: Splitwise, DistServe, 2026 PPD, Prefill-as-a-Service, SplitZip을 하나의 축으로 비교하면 최신 serving 흐름이 선명해집니다.
  • Sequence Parallelism 비교 페이지: Ulysses, Ring Attention, USP, AutoSP를 all-to-all/ring/compiler 관점으로 비교하면 long-context 학습 이해에 좋습니다.
  • Config 선택 실습: “70B를 8×GPU 또는 2노드에 올릴 때 TP/PP/FSDP/CP를 어떻게 고를까?”를 계산 예제로 만들면 학습 페이지가 실전형이 됩니다.
